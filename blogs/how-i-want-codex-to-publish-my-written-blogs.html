How I Want Codex to Publish My Written Blogs

The writing should start in a document. The translation to HTML should stay mechanical.

I do not want to write directly in HTML. I want to think in a document, drop the supporting images into a folder, and let the final publishing step be consistent.

That means the source material should stay raw. The website formatting should be applied later.

What goes into the input folder

Each draft should begin inside a folder under blogs/input/. A simple pattern is blogs/input/blog1/, blogs/input/blog2/, and so on.

  • One main .docx draft with the full article.
  • Any images that actually support the argument.
  • Optional notes or references if they help clarify structure.

The document should carry the thinking. The folder should carry the assets. Nothing in that folder should try to look like the final site.

How the draft itself should be written

The draft should be strong before formatting ever starts. If the document is vague, the generated post will be vague too.

  • Lead with one clear idea, not a topic cloud.
  • Keep paragraphs short enough to survive on a narrow reading column.
  • Use section breaks when the thought changes, not every time a heading sounds nice.
  • Only include images that add evidence, structure, or texture.
  • Write captions or notes in the document if the image needs context.
  • Prefer precise language over broad claims.

A good draft is already shaped. Codex should not have to invent the argument.

Rules for the generated post

The output should match the site that already exists. That means reuse first, and only add new styling when the article genuinely needs it.

  • Create the final post as blogs/<slug>.html.
  • Reuse the shared page shell: head, nav, footer, and ../assets/site.css.
  • Use relative paths only.
  • If images are needed, place published assets in blogs/media/<slug>/ and reference them from the post.
  • Keep the visual style aligned with the existing notes and blog pages.
  • Do not create a one-off design system for a single post.
  • Update blog.html and index.html so the new post appears in site listings.
  • Keep the writing calm, spaced out, and readable on first pass.

The point is not to automate taste. It is to automate the repetitive part after the taste is already present in the draft.

The master prompt

This is the prompt I would give Codex after opening one of those input folders. 

You are inside a blog input folder for my static personal website.

Your job is to convert the source material in this folder into a finished blog post that matches the existing site design.

Context:
- The website is a fully static HTML/CSS site.
- Existing published posts live in /blogs/*.html.
- Shared styling lives in /assets/site.css.
- The post should visually match the current Notes and Blogs pages: minimal, typography-first, calm, no cards, no gradients, no unnecessary effects.
- Use relative paths only.

What to read from this folder:
- The main .docx draft
- Any images in this folder
- Any supporting notes if present

What to do:
1. Read the .docx and extract the real structure of the piece:
   - title
   - subtitle or framing line if one is implied
   - sections
   - captions or image context if present
2. Create a clean final slug for the post.
3. Generate the final article as /blogs/<slug>.html using the same shell as the existing blog posts:
   - same head structure
   - same nav
   - same footer
   - same shared stylesheet reference
4. If images are useful, publish them under /blogs/media/<slug>/ and reference them with relative paths.
5. If the article needs any small shared styling additions, update /assets/site.css minimally instead of creating one-off inline styles.
6. Update /blog.html and /index.html so the new article appears in the Writing / Posts lists with:
   - title
   - date
   - one short description

Writing rules:
- Do not rewrite the piece into generic blog language.
- Preserve the core voice and thinking from the draft.
- Keep paragraphs short.
- Prefer clear section breaks.
- Do not add hype, filler, or marketing language.
- Do not invent claims that are not supported by the draft.
- If the draft is incomplete, make the lightest possible editorial decisions and keep the post honest.

Design rules:
- Match the existing site typography and spacing.
- No frameworks.
- No inline styles.
- No visual clutter.
- Keep image usage restrained and purposeful.

Before finishing:
- Verify all local links resolve.
- Verify image paths are correct.
- Verify the new post is reachable from both /blog.html and /index.html.
- Keep the final output ready to commit as part of the static site.

Why this workflow is worth keeping

It separates thinking from formatting. That is the part I care about.

The draft can stay focused on the idea. Codex can handle the last mile. The site stays consistent without forcing every post to begin as hand-written markup.
